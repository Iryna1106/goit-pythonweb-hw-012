Огляд
=====

Архітектура
-----------

Застосунок побудований на трьох шарах:

* **Шар маршрутів** (:mod:`src.api`) — приймає HTTP-запити, валідує
  payload через Pydantic-схеми, делегує бізнес-логіку до репозиторіїв
  та сервісів і будує відповіді.
* **Сервіси** (:mod:`src.services`) — інкапсулюють бічні ефекти:
  аутентифікація / JWT (:mod:`src.services.auth`), кеш Redis
  (:mod:`src.services.cache`), пошта (:mod:`src.services.email`)
  та завантаження файлів у Cloudinary
  (:mod:`src.services.upload_file`).
* **Репозиторії** (:mod:`src.repository`) — тонкий обгортка над
  SQLAlchemy ``Session`` із операціями для :class:`User` та
  :class:`Contact`.

ORM-моделі живуть у :mod:`src.database.models`. Конфігурація
(:mod:`src.conf.config`) зчитується через ``pydantic-settings`` з
``.env`` або змінних середовища.

Кешування поточного користувача
-------------------------------

Залежність :func:`src.services.auth.get_current_user` спочатку
звертається до Redis. У разі промаху йде у БД і кладе результат назад
у кеш на ``REDIS_USER_CACHE_TTL_SECONDS`` секунд. Будь-яка мутація
користувача (підтвердження пошти, зміна аватарки, паролю, ролі)
викликає :func:`src.services.cache.invalidate_user`, тож наступний
запит читає актуальні дані.

Якщо Redis недоступний, усі допоміжні функції кешу повертають
``None`` / no-op — застосунок коректно деградує до прямого читання з БД.

Ролі користувачів
-----------------

Модель :class:`src.database.models.User` має поле ``role`` типу
:class:`src.database.models.UserRole` (``USER`` за замовчуванням).
:func:`src.services.auth.require_admin` — це FastAPI-залежність, що
повертає 403 для всіх крім адмінів. Маршрути, що нею захищені:

* ``PATCH /api/users/avatar`` — лише адмін може замінювати аватар за
  замовчуванням.
* ``GET /api/users/`` — список усіх користувачів.
* ``PATCH /api/users/{id}/role`` — зміна ролі (з захистом від
  самопониження).

Безпечне скидання пароля
------------------------

JWT для скидання пароля містить унікальний ``jti``-claim. Після
успішного підтвердження
:func:`src.services.cache.mark_password_reset_token_used` атомарно
позначає цей JTI як спожитий (``SET NX EX``). Повторна спроба
використати той самий токен повертає **400 Bad Request** ще до
закінчення його TTL, що убезпечує від replay-атак, якщо посилання
випадково потрапило не в ті руки.
